# Stage 04 — Analytics/Event Pipeline, Step 2

This file is one implementation step from the Stage 04 analytics/event pipeline. Each step is intentionally self-contained for agentic execution: it includes the files to create or modify, classes/types to implement, tests to add, and exit criteria for that step.

Global Stage 04 invariants:

- Redirect hot path must not block on analytics persistence.
- Analytics failures must never change redirect response behavior.
- Raw client IP must not be persisted by default.
- Every redirect attempt should be representable as a normalized click event.
- All public interfaces should have doc comments.
- Prefer updating existing equivalent project files over creating duplicate routing, config, or application-composition files.

Namespace guidance:

- Analytics domain code: `url_shortener::analytics`
- Storage adapters: `url_shortener::storage`
- HTTP adapters/handlers: `url_shortener::http`

## Step 04.2 — Non-blocking queue, analytics service, worker, metrics, and redirect hook

### Goal

Wire analytics into the redirect flow without adding persistence requirements yet. This step introduces the non-blocking fast-path facade, bounded queue, background worker skeleton, metrics abstraction, and HTTP redirect instrumentation hook.

The redirect path must continue to work when analytics is disabled, the queue is full, the event builder rejects input, or the worker/repository path fails.

### Dependencies

Requires Step 04.1 domain types:

- `AnalyticsConfig`
- `AnalyticsError`
- `ClickEvent`
- `ClickEventBuilder`

Repository writes may use a fake/mock repository or the in-memory repository from Step 04.3 if it is already available. Do not block this step on SQL persistence.

## Classes and contracts

### `IClickEventQueue`

Required methods, adapted to project style:

```cpp
virtual EnqueueResult TryEnqueue(ClickEvent event) noexcept = 0;
virtual bool TryDequeue(ClickEvent& out) noexcept = 0;
virtual QueueStats GetStats() const noexcept = 0;
```

Required behavior:

- `TryEnqueue` never blocks on a consumer flush or repository write.
- Full queue returns `DroppedFull`.
- Queue depth is approximate but safe.
- Destruction must not leak threads or block indefinitely.

### `BoundedClickEventQueue`

Required behavior:

- Capacity is fixed at construction.
- Capacity zero is invalid.
- Overflow drops the new event, not an existing queued event, unless a different policy is explicitly documented.
- Multi-producer redirect path must be thread-safe.
- Consumer must be able to drain without busy-spinning.

### `AnalyticsService`

Required behavior:

- Main API: `RecordRedirectAttempt(const RedirectEventContext&) noexcept` or equivalent.
- Returns lightweight status/metrics result if useful, but must not throw or propagate failures into redirect handling.
- If `analytics.enabled=false`, it must be a cheap no-op.
- Builder failures, queue overflow, and metrics failures must not escape.
- Queue overflow increments `analytics_events_dropped_total`.
- Successful enqueue increments `analytics_events_enqueued_total`.

### `AnalyticsWorker`

Required behavior:

- Owns no HTTP state.
- Depends only on `IClickEventQueue`, `IClickEventRepository`, metrics, and config.
- Flushes when batch size is reached.
- Flushes partial batch on interval.
- Retry behavior is bounded by config.
- Persistent repository failure affects analytics only and increments `analytics_worker_failures_total`.
- Stop is graceful and deterministic in tests.

### `RedirectAnalyticsHook`

Required behavior:

- Called after redirect outcome is known.
- Extracts request metadata with minimal allocations.
- Does not know concrete queue or repository types.
- Calls only `AnalyticsService`.
- Must be safe when headers are missing.
- Must never change redirect response status, headers, or body.

## Exit criteria

Step 04.2 is complete when:

- Redirect handler invokes analytics for success and terminal failure paths.
- Redirect behavior is unchanged when analytics is disabled.
- Redirect behavior is unchanged when queue is full.
- Redirect behavior is unchanged when repository/worker fails.
- Queue is bounded by config and has deterministic overflow behavior.
- Worker can batch, flush, retry within bounds, and stop gracefully.
- All listed C++ unit tests pass.
- C++ integration tests for redirect success, failure isolation, queue saturation, and graceful shutdown pass or are explicitly skipped only when the required test harness does not exist yet.
- No SQL/database implementation is required to pass this step.
