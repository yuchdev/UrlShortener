# Stage 04 — Analytics/Event Pipeline, Step 3

This file is one implementation step from the Stage 04 analytics/event pipeline. Each step is intentionally self-contained for agentic execution: it includes the files to create or modify, classes/types to implement, tests to add, and exit criteria for that step.

Global Stage 04 invariants:

- Redirect hot path must not block on analytics persistence.
- Analytics failures must never change redirect response behavior.
- Raw client IP must not be persisted by default.
- Every redirect attempt should be representable as a normalized click event.
- All public interfaces should have doc comments.
- Prefer updating existing equivalent project files over creating duplicate routing, config, or application-composition files.

Namespace guidance:

- Analytics domain code: `url_shortener::analytics`
- Storage adapters: `url_shortener::storage`
- HTTP adapters/handlers: `url_shortener::http`

---

## Step 04.3 — Analytics repository, SQL migration, aggregation, and stats endpoint

### Goal

Persist click events and expose aggregate statistics through the HTTP API. This step turns the queued event pipeline into queryable analytics.

### Dependencies

Requires:

- Step 04.1 domain models: `ClickEvent`, `AggregateQuery`, `AggregateStats`, `AnalyticsError`
- Step 04.2 worker/service/queue pipeline and `IClickEventRepository`

## Classes and contracts

### `IClickEventRepository`

Required contract:

```cpp
virtual Result<void, AnalyticsError> InsertBatch(const std::vector<ClickEvent>& events) = 0;
virtual Result<AggregateStats, AnalyticsError> GetAggregateStats(const AggregateQuery& query) = 0;
virtual Result<uint64_t, AnalyticsError> DeleteOlderThan(Timestamp cutoff) = 0;
```

Required behavior:

- Empty batch is a no-op success.
- Batch insert must be atomic per backend if practical.
- Repository must never persist raw IP because `ClickEvent` must not contain it.
- Aggregation must filter by slug/link and time range.
- Aggregation must return total attempts, successful redirects, counts by status code, counts by domain, and time buckets.

### `InMemoryClickEventRepository`

Required behavior:

- Deterministic and thread-safe enough for tests.
- Maintains inserted events in memory.
- Implements the same aggregation semantics as SQL repository.
- Supports retention delete by timestamp cutoff.
- Useful as local/test backend, not as production durability backend.

### `SqlClickEventRepository`

Required behavior:

- Supports SQLite for local/integration tests.
- Supports PostgreSQL when the project CI has a database service enabled.
- Uses prepared statements/bound parameters.
- Adds indexes for common queries:
  - `(slug, occurred_at)`
  - `(slug, status_code)` if useful
  - `(slug, domain)` if useful
- Keeps SQL dialect branching isolated.
- Uses UTC timestamps consistently.

### `AnalyticsStatsHandler`

Required behavior:

Endpoint:

```http
GET /api/v1/links/{slug}/stats?from=<ts>&to=<ts>&bucket=day
```

Response minimum:

```json
{
  "slug": "abc123",
  "link_id": "optional-internal-id",
  "total_attempts": 10,
  "successful_redirects": 8,
  "attempts_by_status_code": { "302": 8, "404": 2 },
  "attempts_by_domain": { "sho.rt": 10 },
  "time_buckets": [
    { "bucket_start": "2026-05-15T00:00:00Z", "count": 10 }
  ],
  "from": "2026-05-01T00:00:00Z",
  "to": "2026-05-16T00:00:00Z",
  "bucket": "day"
}
```

Validation behavior:

- Invalid timestamp => `400`.
- Invalid range => `400`.
- Unsupported bucket => `400`.
- Unknown slug => choose one project policy and document it:
  - preferred: `404`, or
  - acceptable: `200` with zero counts if existing product API style uses empty results.

## Exit criteria

Step 04.3 is complete when:

- In-memory repository implements insert, aggregate, and retention deletion semantics.
- SQL repository roundtrip passes for SQLite.
- PostgreSQL repository roundtrip passes when PostgreSQL CI service is enabled, or is explicitly skipped with a clear marker when unavailable.
- Worker from Step 04.2 can persist events through `IClickEventRepository`.
- Stats endpoint returns correct totals, status-code counts, domain counts, and time buckets.
- Stats endpoint validates bad input and returns project-standard error responses.
- Python black-box schema/validation tests pass against the compiled server/test harness.
- No raw IP appears in persisted event data.
