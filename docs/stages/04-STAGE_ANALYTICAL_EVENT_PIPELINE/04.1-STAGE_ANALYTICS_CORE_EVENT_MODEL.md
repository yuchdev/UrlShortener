# Stage 04 — Analytics/Event Pipeline

This file is one implementation step from the Stage 04 analytics/event pipeline. Each step is intentionally self-contained for agentic execution: it includes the files to create or modify, classes/types to implement, tests to add, and exit criteria for that step.

Global Stage 04 invariants:

- Redirect hot path must not block on analytics persistence.
- Analytics failures must never change redirect response behavior.
- Raw client IP must not be persisted by default.
- Every redirect attempt should be representable as a normalized click event.
- All public interfaces should have doc comments.
- Prefer updating existing equivalent project files over creating duplicate routing, config, or application-composition files.

Namespace guidance:

- Analytics domain code: `url_shortener::analytics`
- Storage adapters: `url_shortener::storage`
- HTTP adapters/handlers: `url_shortener::http`

---

## Step 04.1 — Core event model, sanitization, anonymization, and query DTOs

### Goal

Create the analytics domain foundation without wiring it into redirect handling yet. This step defines safe data structures and deterministic helpers used by all later steps.

### Deliverables

By the end of this step, the repository contains a compilable analytics domain module with:

- analytics configuration defaults and validation
- analytics error model
- `ClickEvent` model
- request metadata sanitization/truncation helpers
- client identifier hashing helper
- `ClickEventBuilder`
- aggregate query/response DTOs
- unit tests proving validation, privacy, and deterministic truncation rules

No HTTP route, worker thread, queue, or database table is required in this step.

## Classes and contracts

### `AnalyticsConfig`

Required fields:

```cpp
bool enabled = true;
std::size_t queue_capacity = 8192;
std::size_t batch_size = 128;
std::chrono::milliseconds flush_interval = std::chrono::milliseconds(1000);
AnalyticsRetryConfig retry;
AnalyticsRetentionConfig retention;
SanitizationLimits sanitization;
std::string client_hash_salt;
```

Required behavior:

- `Validate()` returns `Result<void, AnalyticsError>` or equivalent project validation result.
- Invalid queue capacity, batch size, negative flush interval, empty salt in production mode, and invalid sanitization limits must be rejected.
- Defaults must be safe for local development.
- Salt may be provided by config or environment override in a later composition step, but this class must already support the value.

### `ClickEvent`

Required behavior:

- Represents one redirect attempt/outcome.
- Allows successful redirect statuses and terminal statuses such as 404/410.
- Must not expose or persist a raw IP field.
- Optional strings may be empty but must already be sanitized before storage.
- Should be cheap to move.

### `ClickEventSanitizer`

Required behavior:

- `referrer`: max 512 bytes by default.
- `user_agent`: max 512 bytes by default.
- `domain`: max 255 bytes by default.
- Strip ASCII control characters and normalize surrounding whitespace.
- Truncation must be deterministic and byte-length safe.
- Do not copy arbitrary request headers.

### `ClientIdHasher`

Required behavior:

- Same normalized client identifier + same salt => same hash.
- Same identifier + different salt => different hash.
- Empty input returns empty/anonymous hash only if the config explicitly allows missing network identifier; otherwise validation error.
- Output must be hex or base64 digest, optionally truncated by config.
- Output must never equal the input raw IP.

### `ClickEventBuilder`

Required behavior:

- Accepts `RedirectEventContext` with fields such as slug, link id, domain, final status code, referrer, user agent, client network identifier, and timestamp source.
- Produces a validated and sanitized `ClickEvent`.
- Missing referrer and user-agent become empty optional fields, not errors.
- Invalid slug/domain/status should return an error rather than creating a broken event.

### `AggregateQueryValidator`

Required behavior:

- Accept `hour`, `day`, and optionally `week` if simple to support consistently.
- Reject `from > to`.
- Reject unsupported buckets.
- Define default bucket when missing, preferably `day`.
- Reject excessive ranges if the project has existing anti-abuse validation policy.

## Exit criteria

Step 04.1 is complete when:

- All files listed above exist or equivalent existing files are updated and documented in the PR notes.
- Analytics domain code compiles as part of the main project.
- All five required C++ unit tests pass.
- The `ClickEvent` model has no raw IP field.
- Sanitization limits are deterministic and covered by tests.
- Hashing is deterministic for same salt and changes with salt rotation.
- No redirect path, background worker, repository persistence, or HTTP endpoint behavior is changed yet.
